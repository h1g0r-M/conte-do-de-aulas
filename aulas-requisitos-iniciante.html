Conceito básico

O que é um Requisito?

💡 Em palavras simples

Um requisito é tudo aquilo que o sistema precisa fazer ou precisa ser para satisfazer quem vai usá-lo.

🏠

Antes de construir uma casa, você lista o que ela precisa ter: 2 quartos, 1 banheiro, garagem, cozinha. Isso são os requisitos da casa. No software é a mesma coisa!

🤔

Por que os requisitos são importantes?

  • Sem eles, ninguém sabe o que construir
  • Evitam que o sistema seja feito "errado"
  • Evitam retrabalho e desperdício de dinheiro
  • São a base para testar se o sistema funciona
📌 Dado curioso

Estudos mostram que a maioria dos projetos de software falha por causa de requisitos mal definidos — não por problemas técnicos!

Visão geral

Existem dois tipos principais

⚙️

Requisito Funcional (RF)

O que o sistema faz.
São as funções, ações e serviços que o software oferece ao usuário.

"O sistema deve permitir..."
"O sistema deve calcular..."
"O sistema deve enviar..."
🛡️

Requisito Não Funcional (RNF)

Como o sistema se comporta.
São as qualidades que o sistema precisa ter: velocidade, segurança, facilidade de uso.

"O sistema deve responder em..."
"O sistema deve estar disponível..."
"O sistema deve ser seguro..."
🎓 Resumindo

RF = O que faz  |  RNF = Como funciona
Ex: "O sistema deve fazer login" (RF)  |  "O login deve ser feito em menos de 2 segundos" (RNF)

Aprofundando

Requisitos Funcionais — RF

Um RF descreve uma ação ou função que o sistema precisa executar. Sempre começa com:
"O sistema deve [verbo]..."

  • Registrar um novo usuário
  • Calcular o total de uma compra
  • Emitir uma nota fiscal
  • Enviar um e-mail de confirmação
  • Bloquear uma conta após erros de senha
💡 Dica

Use sempre um verbo de ação: calcular, enviar, registrar, bloquear, exibir, gerar, permitir, validar...

📱

Exemplo: App de Delivery

  • O sistema deve mostrar o cardápio do restaurante
  • O sistema deve adicionar itens ao carrinho
  • O sistema deve calcular o valor do pedido
  • O sistema deve rastrear o entregador em tempo real
  • O sistema deve notificar quando o pedido sair para entrega
Na prática

Como escrever um RF corretamente

❌ RF mal escrito

"O sistema deve ser bom no login."

  • O que é "ser bom"? Cada pessoa entende diferente
  • Impossível saber se está pronto
  • Gera brigas entre cliente e desenvolvedor
✅ RF bem escrito

"O sistema deve permitir que o usuário faça login com e-mail e senha cadastrados."

  • Claro e objetivo
  • Dá para testar: funciona ou não funciona
  • Todo mundo entende do mesmo jeito
Fórmula simples
"O sistema deve" + [verbo de ação] + [o quê] + [para quem / em qual situação]
Organização

Nem todo requisito tem a mesma importância

Os RFs precisam ser priorizados para que a equipe saiba o que desenvolver primeiro.

🔴

Essencial

O sistema não funciona sem isso. É obrigatório!


Ex: Um banco precisa conseguir transferir dinheiro.

🟡

Importante

Melhora muito o sistema, mas dá para lançar sem ele.


Ex: Histórico de transferências do mês.

🟢

Desejável

Seria legal ter, mas não é urgente. Pode vir depois.


Ex: Gráfico de gastos por categoria.

💡 Dica prática

Sempre comece pelos essenciais. Um sistema simples que funciona é melhor que um sistema cheio de funções que não funciona direito!

✏️ Exercício — Aula 1
🏫 Situação: Você foi chamado para ajudar a criar um sistema de chamada escolar. O professor precisa marcar a presença dos alunos pelo celular.
1

Escreva 4 Requisitos Funcionais para esse sistema. Use a fórmula: "O sistema deve [verbo]..."

2

Coloque uma prioridade para cada RF: Essencial Importante Desejável

3

O RF abaixo está bem escrito? Por quê?
"O sistema deve funcionar bem para o professor."

✅ RF-01 [Essencial]: O sistema deve exibir a lista de alunos da turma selecionada.
✅ RF-02 [Essencial]: O sistema deve permitir que o professor marque presença ou falta para cada aluno.
✅ RF-03 [Importante]: O sistema deve gerar relatório de frequência por aluno no mês.
✅ RF-04 [Desejável]: O sistema deve enviar aviso aos pais quando o aluno tiver mais de 3 faltas.

❌ O RF "funcionar bem" está mal escrito — é vago e impossível de testar. O correto seria: "O sistema deve registrar a chamada em até 5 segundos por turma."
Aula 2 de 4

Requisitos Não Funcionais

A qualidade que ninguém vê — mas todo mundo sente quando está faltando.

⏱️ 60 minutos
🎯 Nível iniciante
✏️ Exercício incluído
🛡️
Conceito básico

O que são Requisitos Não Funcionais?

💡 Em palavras simples

Um RNF não diz o que o sistema faz, mas como ele faz: com que velocidade, com que segurança, com que facilidade.

🚗

Pedir um carro com ar-condicionado, direção hidráulica e 0 a 100km em 8 segundos são requisitos não funcionais. Não mudam o que o carro faz (transportar pessoas), mas mudam como faz isso.

⚠️

Por que são tão importantes?

  • Um sistema lento frustra o usuário, mesmo funcionando
  • Um sistema inseguro pode vazar dados de todos
  • Um sistema difícil de usar é abandonado
  • Um sistema que "cai" o tempo todo não serve para nada
📌 Comparação rápida

RF: "O sistema deve processar pagamentos"  →  RNF: "O pagamento deve ser concluído em menos de 3 segundos"

Categorias

Os 5 tipos mais comuns de RNF

  • ⚡ Desempenho
    Quão rápido o sistema responde.
    Ex: A busca deve retornar resultados em até 2 segundos.
  • 🔒 Segurança
    Proteção dos dados e do acesso.
    Ex: Senhas devem ficar criptografadas no banco de dados.
  • 🟢 Disponibilidade
    Quanto tempo o sistema fica no ar.
    Ex: O sistema deve funcionar 24h por dia, 7 dias por semana.
  • 😊 Usabilidade
    Facilidade de uso pelo usuário.
    Ex: Um usuário novo deve conseguir fazer um pedido em menos de 5 minutos.
  • 📱 Compatibilidade
    Funcionar em diferentes dispositivos e sistemas.
    Ex: O site deve funcionar no celular, tablet e computador.
📊

Resumo visual

TipoPergunta-chave
⚡ DesempenhoQuão rápido?
🔒 SegurançaQuão seguro?
🟢 DisponibilidadePor quanto tempo?
😊 UsabilidadeQuão fácil?
📱 CompatibilidadeEm quais lugares?
Comparação

RF e RNF — Lado a Lado

Veja como os dois tipos de requisito se complementam no mesmo sistema:

Sistema 🟠 Requisito Funcional (o que faz) 🔵 Requisito Não Funcional (como faz)
🛒 E-commerce O sistema deve finalizar a compra com cartão de crédito O pagamento deve ser confirmado em até 5 segundos
🏦 Banco O sistema deve permitir transferência Pix O Pix deve ser processado 24h por dia, 365 dias por ano
📚 Escola O sistema deve emitir boletim do aluno O boletim deve poder ser acessado de qualquer celular
🏥 Clínica O sistema deve registrar prontuário do paciente Os dados do paciente devem ser criptografados (LGPD)
🎓 Lembre-se

Para cada RF importante, vale perguntar: "Como esse requisito deve se comportar?" A resposta vira um RNF!

Na prática

Como escrever um RNF que presta

O segredo é usar números e medidas. Sem isso, o RNF não tem valor.

❌ RNF RUIM — vago

"O sistema deve ser rápido."

❌ RNF RUIM — vago

"O sistema deve ser seguro."

✅ RNF BOM — mensurável

"O sistema deve responder a qualquer pesquisa em menos de 3 segundos."

✅ RNF BOM — mensurável

"As senhas devem ter no mínimo 8 caracteres com letras e números."

Se você não consegue medir, não é um bom requisito!
Mundo real

RNF no mundo real — Você já sentiu isso!

🐢

Desempenho ruim

Já ficou esperando um site carregar por mais de 10 segundos e desistiu? Isso é um RNF de desempenho não atendido.

😵

Usabilidade ruim

Já entrou num app e não conseguiu achar como fazer o que queria? Isso é um RNF de usabilidade não atendido.

📵

Disponibilidade ruim

Já tentou acessar um sistema e ele estava "fora do ar"? Isso é um RNF de disponibilidade não atendido.

💡 Conclusão

Os RNFs são o que fazem o usuário adorar ou odiar um sistema. Um sistema pode ter todas as funções certas e ainda assim ser inutilizável por causa de RNFs ruins!

✏️ Exercício — Aula 2
🏦 Situação: Você está levantando requisitos de um aplicativo de banco digital. Já foram definidos os Requisitos Funcionais. Agora é hora de pensar nos Não Funcionais.
1

Escreva 1 RNF para cada tipo: Desempenho, Segurança e Disponibilidade. Use medidas concretas (números, porcentagem, tempo).

2

O RNF abaixo está bom ou ruim? Reescreva-o melhor:
"O app deve ser seguro para os usuários."

3

Na sua opinião, qual o RNF mais importante para um banco digital? Explique com suas palavras.

✅ Desempenho: "As transações Pix devem ser concluídas em até 10 segundos."
✅ Segurança: "O app deve exigir confirmação biométrica (digital ou rosto) para cada transação."
✅ Disponibilidade: "O app deve funcionar 24 horas por dia, todos os dias do ano."

❌ "Seguro para os usuários" está vago. Melhor: "O app deve bloquear o acesso após 3 tentativas erradas de senha e notificar o usuário por e-mail."

🏆 O mais importante: Segurança — porque envolve dinheiro real das pessoas. Qualquer falha pode causar prejuízo financeiro grave.
Aula 3 de 4

Como Descobrir os Requisitos?

Técnicas simples para levantar o que o sistema realmente precisa fazer.

⏱️ 60 minutos
🎯 Nível iniciante
✏️ Exercício incluído
🔎
O desafio

O problema: ninguém sabe exatamente o que quer!

💬 Situação real

O cliente diz: "Quero um sistema de vendas."

Mas o que ele realmente quer? Controle de estoque? Emissão de nota fiscal? Relatórios? Integração com o delivery?

💇

É como ir ao cabeleireiro e dizer "corta um pouco". Cada cabeleireiro vai entender diferente! A tarefa do analista é descobrir exatamente o que o cliente precisa.

🕵️

O que é elicitação?

É o nome bonito para o processo de descobrir e levantar os requisitos junto com quem vai usar o sistema.

  • Elicitar = extrair, descobrir, levantar
  • Feita pelo Analista de Requisitos
  • Usa diferentes técnicas dependendo do contexto
Passo 1

Primeiro: identifique quem consultar

Essas pessoas se chamam stakeholders — todos que têm interesse no sistema.

👤

Usuário Final

Quem vai usar o sistema no dia a dia. Sabe o que precisa na prática.

Ex: Caixa, secretária, vendedor

👔

Gestor / Cliente

Quem pediu o sistema e paga pelo projeto. Sabe os objetivos do negócio.

Ex: Diretor, gerente, dono

⚙️

Equipe Técnica

Desenvolvedores, DBA. Sabem o que é possível fazer tecnicamente.

Ex: Programador, arquiteto

💡 Dica de ouro

Cada stakeholder tem uma visão diferente do sistema. O segredo é ouvir todos e encontrar o que realmente importa para o projeto funcionar.

Técnica 1

Entrevista — Conversar para descobrir

📌 O que é?

Uma conversa com o stakeholder para entender o que ele precisa. É a técnica mais usada e mais poderosa!

  • Pode ser presencial ou por videochamada
  • Pode ter roteiro (perguntas planejadas) ou ser livre
  • Melhor para entender bem a fundo
  • Mais demorada que outras técnicas
⚠️ Cuidado!

Prefira perguntas abertas ("Me conte como funciona...") do que fechadas ("Você usa relatório diário?"). Perguntas abertas revelam muito mais!

💬

Exemplos de boas perguntas

  • "Me conte como você faz esse processo hoje, do início ao fim."
  • "O que te incomoda mais no jeito que as coisas funcionam hoje?"
  • "O que você espera que o sistema novo faça por você?"
  • "Existe alguma situação difícil que o sistema precisa resolver?"
  • "Quem mais usa esse processo além de você?"
Técnica 2 e 3

Questionário e Observação

📋

Questionário

Um formulário com perguntas enviado para muitas pessoas ao mesmo tempo (Google Forms, por exemplo).

  • Ótimo quando há muitos usuários para consultar
  • Rápido e barato de aplicar
  • Não permite aprofundar as respostas
Ideal para: validar ideias com muita gente de uma vez só.
👁️

Observação

O analista vai até onde o usuário trabalha e observa como ele faz as tarefas no dia a dia.

  • Revela o que o usuário não sabe descrever
  • Captura detalhes que ninguém menciona
  • O usuário às vezes faz diferente do que fala!
Ideal para: processos complexos do cotidiano.
💡 Exemplo de observação

Uma recepcionista disse: "Faço tudo no sistema". Mas ao ser observada, a analista descobriu que ela usava 3 post-its colados no monitor para lembrar informações. Isso virou um requisito de "memorização de favoritos" no sistema!

Técnica 4 e 5

Reunião em Grupo e Protótipo

🧩

Workshop / Reunião em Grupo

Uma reunião com várias pessoas ao mesmo tempo: usuários, gestores e analistas juntos para levantar e alinhar os requisitos.

  • Resolve conflitos na hora
  • Todos chegam a um acordo juntos
  • Gera muitos requisitos em pouco tempo
  • Precisa de um facilitador habilidoso
🖼️

Prototipação (Tela Rascunho)

Criar um "esboço" de como o sistema vai ser — pode ser no papel mesmo! — para o usuário ver e validar.

  • O usuário "vê" o sistema antes de ser feito
  • Aponta erros antes de programar
  • Pode ser feito no papel, no Figma ou no PowerPoint
  • Economiza muito tempo e dinheiro
💡 Dica prática

Na maioria dos projetos, a melhor estratégia é combinar: Entrevista para entender → Workshop para alinhar → Protótipo para confirmar!

✏️ Exercício — Aula 3
🍕 Situação: Uma pizzaria quer criar um sistema de pedidos online. Hoje tudo é feito por telefone e há muita confusão com os pedidos errados.
1

Liste os stakeholders que você precisaria consultar. Quem são eles e o que cada um sabe?

2

Escolha 2 técnicas de coleta que usaria nesse projeto e explique por quê.

3

Escreva 4 perguntas de entrevista que você faria ao atendente da pizzaria. Use perguntas abertas.

👥 Stakeholders: Dono da pizzaria (objetivos do negócio), atendente (como funciona hoje), entregador (fluxo de entrega), cliente (o que espera ao pedir), cozinheiro (como recebe os pedidos).

🔧 Técnicas: Observação (para ver como o atendente lida com os pedidos no telefone hoje) + Entrevista (para entender as dores de cada pessoa).

💬 Perguntas de entrevista:
1. "Me descreva como funciona o processo de receber um pedido, do início ao fim."
2. "Quais são os erros mais comuns que acontecem hoje nos pedidos?"
3. "O que você precisa saber do cliente para registrar um pedido corretamente?"
4. "O que você mais sentiria falta se o telefone sumisse e tivesse que usar só o sistema?"
Aula 4 de 4

Documentando e Validando

Como organizar e confirmar os requisitos — fechando o ciclo de análise.

⏱️ 60 minutos
🎯 Nível iniciante
✏️ Exercício final
📝
Importância

Por que documentar os requisitos?

📌 Situação real

Imagine que você levantou todos os requisitos com o cliente, mas não anotou nada. 3 meses depois, o sistema está pronto — mas o cliente diz que "não era bem isso que eu queria".

💡 A documentação resolve isso

Com o documento de requisitos assinado, todos concordaram e registraram o que deveria ser feito. É a proteção do desenvolvedor e do cliente!

📄

O que a documentação garante?

  • Todos sabem o que precisa ser feito
  • Novos membros da equipe entendem o projeto
  • Base para criar os testes do sistema
  • Referência quando surgem dúvidas
  • Prova do que foi acordado com o cliente
Estrutura

Como organizar um requisito documentado

Cada requisito deve ter informações básicas para ser útil e rastreável.

CampoO que significa
CódigoUm número único: RF-01, RNF-02...
NomeTítulo curto do requisito
DescriçãoO que o sistema deve fazer (com detalhes)
PrioridadeEssencial / Importante / Desejável
OrigemQuem pediu esse requisito
Critério de aceiteComo saber que está pronto?
📋

Exemplo de RF documentado

CódigoRF-03
NomeRecuperar senha
DescriçãoO sistema deve enviar um link para o e-mail do usuário redefinir a senha
PrioridadeEssencial
OrigemEntrevista com usuários (João, Maria)
AceiteLink chega em até 2 min e expira em 30 min
Validação

Validar: confirmar que está certo

📌 O que é validação?

É o processo de confirmar com o cliente e os usuários que os requisitos levantados realmente refletem o que eles precisam.

🍽️

É como pedir ao garçom que repita o pedido antes de ir à cozinha: "Então, você quer um filé ao ponto, sem cebola, com fritas?". Confirmar evita erros!

✔️

Como validar?

  • Revisão com o cliente: apresentar a lista de requisitos e confirmar cada um
  • Protótipo: mostrar um esboço do sistema para o usuário aprovar
  • Casos de uso: contar como o sistema vai ser usado passo a passo
  • Assinatura: o cliente assina o documento confirmando que está de acordo
Atenção

Erros comuns de quem está começando

  • Requisitos vagos
    "O sistema deve ser bom" → Como saber se ficou bom?
  • Assumir sem perguntar
    Inventar o que o cliente precisa sem verificar.
  • Não documentar
    Combinar verbalmente e esquecer de registrar.
  • Ignorar os RNFs
    Focar só no que o sistema faz e esquecer a qualidade.
  • Consultar só uma pessoa
    O dono sabe o negócio, mas o usuário sabe o dia a dia.
🏆

Boas práticas para iniciantes

  • Sempre anote tudo durante as entrevistas
  • Releia os requisitos e pergunte: "É possível testar isso?"
  • Confirme com o cliente antes de passar para o desenvolvimento
  • Numere cada requisito (RF-01, RF-02...)
  • Defina uma prioridade para cada requisito

🎓 Situação: Uma escola técnica quer um sistema de gestão de estágios. Os alunos precisam registrar horas de estágio, os professores orientadores precisam acompanhar, e a coordenação precisa de relatórios.
1

Escreva 3 Requisitos Funcionais e 2 Requisitos Não Funcionais para esse sistema. Use código (RF-01, RNF-01) e prioridade.

2

Escolha 1 técnica de coleta e diga com qual stakeholder a usaria e por quê.

3

Preencha a ficha de um requisito completo (código, nome, descrição, prioridade, origem e critério de aceite) para um dos seus RFs.

✅ RF-01 [Essencial]: O sistema deve permitir que o aluno registre as horas de estágio diariamente.
✅ RF-02 [Essencial]: O sistema deve exibir ao professor orientador o total de horas registradas por cada aluno.
✅ RF-03 [Importante]: O sistema deve gerar relatório em PDF com o histórico de horas do aluno.
✅ RNF-01 [Essencial]: O sistema deve funcionar no celular (Android e iPhone).
✅ RNF-02 [Importante]: O registro de horas deve ser salvo em até 3 segundos.

🔧 Técnica: Entrevista com o aluno — porque ele é quem vai usar o sistema no dia a dia e sabe o que precisa.

📋 Ficha do RF-01:
Código: RF-01 | Nome: Registro de horas | Descrição: O sistema deve permitir ao aluno registrar a data, horário de entrada e saída do estágio.
Prioridade: Essencial | Origem: Entrevista com alunos | Aceite: Registro salvo com sucesso e visível para o orientador.
